# AC-012 — Performance de carga inicial de la app mobile

> Tiempo entre el tap del usuario en el icono de la app y la primera pantalla interactuable (RF-017 CTA bifurcado o T-033 "Mis tasaciones" según token).

## Planguage

| Slot | Valor |
|------|-------|
| **Nombre del atributo** | tiempo-carga-inicial-app-mobile |
| **Categoría** | Performance |
| **Escala** | Segundos transcurridos cronómetro desde el tap del usuario en el icono de la app en su launcher (Android / iOS) hasta que la primera pantalla (RF-017 CTA o T-033 home según haya token JWT) es **completamente interactuable** (touches funcionan, no hay spinner bloqueante). |

## Verificación binaria

✓ Cumple si p95 ≤ 3 segundos sobre 100 cold-starts en producción durante la semana del Hito 1.
✗ No cumple si p95 > 3 segundos.

## Por qué importa

- En contexto de tasador en campo (caso de uso primario), una app que tarda > 3s en abrir genera fricción real.
- Para el Cliente B2C (autotasador), una primera impresión lenta genera bounce: si la app tarda mucho, el cliente abandona antes de empezar.
- Afecta especialmente al criterio 10/10 (BR-NEG-002): si la app es lenta en la demo, los 10 arquitectos quedan con mala primera impresión.

## Tradeoffs

- **AC-005 Compliance (encriptación at-rest)**: overhead mínimo (≤ 50ms), no debería impactar.
- **Stack inicial pesado** (React Native / Flutter / nativo) afecta este AC. Decisión técnica DP-007 (stack final) **debe contemplar** el tiempo de cold start como criterio de elección.

## Riesgo

- Bundle size grande (>10 MB) infla el cold start en Android. Mitigación: code-splitting + assets optimizados + lazy loading de features no críticas.
- Latencia del JWT validation al primer request degrada percepción de "interactivo". Mitigación: validación local primero (verificar expiración del token sin llamada al backend), revalidar contra server async.

## Mediciones recomendadas

- Firebase Performance / Sentry Performance configurado desde el primer build deployado a producción.
- Alerta automática si p95 > 3s durante la semana del Hito 1.

## Fuera del MVP-6sem

- Performance budget formal con CI/CD que bloquee merges si degrada.
- Optimización de **Time to Interactive** (TTI) específico de cada pantalla (no solo cold start de la app).
- Warm start metric.
