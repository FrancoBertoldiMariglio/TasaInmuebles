# ADR-001: App Mobile única bifurcada por rol (tasador + cliente B2C)

## Estado

Aceptada

Fecha: 2026-05-15
Reemplaza: —

## Contexto

El MVP debe soportar dos flujos de usuario simultáneos sobre el mismo dispositivo móvil: el tasador profesional (relevamiento en campo + comité) y el cliente B2C (autotasación referencial). La decisión 2026-05-14 movió el autotasador de Fase 3 a MVP-6sem, lo que obliga a tomar una decisión arquitectónica explícita sobre la convivencia de ambos flujos.

La sociedad cuenta con un único desarrollador hasta el Hito 1 (Franco). La prioridad de mantenibilidad domina las decisiones.

## Opciones Consideradas

| Opción | Descripción | Pros | Contras |
|---|---|---|---|
| A: App única bifurcada por rol | Una sola aplicación mobile. En el inicio de sesión, CTA bifurcado: "Soy tasador (login)" / "Soy cliente (autoregistro)". Cada rol ve solo su flujo. | Una sola base de código. Pitch al Colegio muestra ambos productos coexistiendo. Reusa componentes (formulario inmueble, cálculo Fitt-Servini, PDF) | Acoplamiento UX entre roles; riesgo de confusión si el branding no es claro |
| B: Dos apps separadas | App profesional + app B2C, distribuidas por separado | UX más enfocada por rol; menor superficie de seguridad en cada app | Doble base de código + doble distribución; equipo de 1 dev no escala |
| C: App profesional + web pública B2C | Profesional mobile, B2C web responsiva | Captación B2C por SEO; separación clara | Web pública implica scope adicional (landing, SEO) no priorizado para Hito 1 |

## Decisión

Se elige la **Opción A — App única bifurcada por rol**, alineada con la prioridad del proyecto (Mantenibilidad > Seguridad > Costo > Performance > Estabilidad).

Razones específicas:

- Una sola base de código reduce la carga del equipo de 1 desarrollador.
- El pitch al Colegio se beneficia de mostrar el producto unificado en demo: "el mismo arquitecto puede usar la app como profesional y como cliente B2C cuando quiera".
- Reusa formulario de inmueble, cálculo Fitt-Servini, generación de PDF y storage entre ambos flujos.
- La separación UX se resuelve con bifurcación en pantalla de bienvenida + flujos completamente independientes una vez logueado.

## Consecuencias

**Positivas**:

- Menor superficie de mantenimiento.
- Reuso de componentes lógicos sin duplicación.
- Demo más impactante al Colegio.

**Negativas**:

- Acoplamiento entre evoluciones de ambos flujos: cambios al modelo de tasación o al cálculo impactan a los dos. Mitigación: tests que cubran ambos caminos críticos.
- Si la base B2C crece mucho en Fase 2 y diverge del flujo profesional, puede ameritar separar. Mitigación: revisitar este ADR al final de Fase 2.
- Vector de seguridad ligeramente más amplio (un usuario malicioso podría intentar manipular el rol). Mitigación: rol único persistido server-side; cliente nunca decide rol.

**Neutras**:

- La distribución de la app (store) no cambia: una sola app, una sola publicación.
