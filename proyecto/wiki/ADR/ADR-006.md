# ADR-006: Monolito modular preferido en MVP; división a microservicios se difiere a Fase 2

## Estado

Aceptada

Fecha: 2026-05-15
Reemplaza: —

## Contexto

El sistema tiene 5 componentes lógicos en el backend (API, Fitt-Servini, Robotomus Mock, Generador PDF, Identidad) más Persistencia y Storage. En Fase 2 se incorporan Robotomus real, scraping, integración Inmoclick, dashboards B2B, módulo contable.

El debate clásico monolito vs microservicios aplica: en MVP con 1 dev y dominio aún no maduro, partir prematuramente penaliza velocidad y operaciones. En Fase 2 con equipo más grande y dominio asentado, dividir puede ser conveniente.

## Opciones Consideradas

| Opción | Descripción | Pros | Contras |
|---|---|---|---|
| A: Monolito modular en MVP, división diferida | Todos los componentes en un solo proceso, organizados en módulos con fronteras claras. Cada módulo expone una interfaz y los demás lo consumen por DI / IoC | Velocidad de desarrollo; debug simple; deploy único; cabe en 1 dev | Acoplamiento de release; un módulo problemático afecta a todos |
| B: Microservicios desde día 1 | Cada componente lógico es un servicio independiente, con API y deploy separados | Escalable por componente; tolerancia a fallos por servicio | Complejidad operativa alta; latencia de red entre componentes; consume tiempo del Hito 1 |
| C: Híbrido (API monolito + Robotomus separado) | Anticipar la separación solo del componente más probable a crecer | Aísla el camino futuro complejo | Introduce complejidad operativa sin justificar; Robotomus es mock en MVP |

## Decisión

Se elige la **Opción A — Monolito modular en MVP**, alineada con la prioridad (Mantenibilidad > Costo > Performance > Estabilidad).

Restricciones operativas para el monolito modular:

- Cada módulo (Fitt-Servini, Robotomus, PDF Gen, Identidad) expone una **interfaz** explícita y se consume vía dependency injection.
- Los módulos no comparten estado en memoria; cualquier dato compartido va por Persistencia o Storage.
- Los módulos no se llaman entre sí circularmente — sólo el API Backend orquesta.
- Tests unitarios por módulo; tests de integración por flujo completo.

Esto deja el sistema **preparado para dividir** cuando Fase 2 lo amerite, sin re-arquitectura.

## Consecuencias

**Positivas**:

- Deploy único, debug local simple, sin orquestación de servicios.
- Latencia interna baja (in-process).
- Cabe en el Hito 1 con margen.
- Refactorizar luego a microservicios es factible si las fronteras se mantienen estrictas.

**Negativas**:

- Acoplamiento de release: un cambio en Fitt-Servini requiere redeploy del monolito. Mitigación: deploys frecuentes con CI/CD; ventana de mantenimiento corta.
- Una falla catastrófica del proceso afecta a todo. Mitigación: health checks + reinicios automatizados; en MVP con N usuarios bajos es tolerable.
- Si un módulo termina muy grande (Robotomus real en Fase 2), puede empujar a separar antes de lo previsto. Mitigación: re-evaluar al inicio de Fase 2.

**Neutras**:

- La elección del stack (DP-S7) puede facilitar o dificultar este patrón (algunos lenguajes/frameworks favorecen monolitos modulares; otros, microservicios). El ADR mantiene la intención independientemente del stack.
- Cuando se divida en Fase 2, los ADRs nuevos referenciarán este como punto de partida.
