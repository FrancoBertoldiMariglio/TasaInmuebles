# ADR-002: Mobile-only en MVP Hito 1; web y back office difieren a Fase 2

## Estado

Aceptada

Fecha: 2026-05-15
Reemplaza: —

## Contexto

El producto contempla múltiples canales según el rol: tasador en campo (mobile-first), cliente B2C (mobile), admin de plataforma, dashboards B2B para owners de bancos / constructoras / inmobiliarias.

El Hito 1 está acotado a ~6 semanas de desarrollo con 1 dev. El piloto del Colegio involucra solo a tasadores y, opcionalmente, a clientes B2C — no a entidades B2B reales todavía.

## Opciones Consideradas

| Opción | Descripción | Pros | Contras |
|---|---|---|---|
| A: Mobile-only en MVP, web a Fase 2 | App mobile para tasador + B2C. Admin operado por CLI / script. Sin web pública ni dashboards B2B | Foco máximo en el flujo crítico del Colegio. Cabe en 6 semanas | No hay vidriera web para captación; admin sin UI |
| B: Mobile + admin web | App mobile + back office web mínimo para admin | Admin con UI desde día 1 | Doble esfuerzo de UI; back office es Fase 2 según roadmap |
| C: Mobile + landing web pública (SEO) | App mobile + landing pública para captación B2C orgánica | Captación post-Hito 1 más rápida | SEO + landing dilatan el Hito 1; no aporta a la demo del Colegio |

## Decisión

Se elige la **Opción A — Mobile-only en MVP**, alineada con la prioridad (Mantenibilidad y Tiempo del Hito 1).

Razones específicas:

- El pitch al Colegio es sobre el flujo profesional del tasador en campo. Una web pública no aporta a esa demo.
- Los ~10 arquitectos pueden operar con la app mobile sin necesidad de versión desktop.
- El admin del MVP es Franco o Cocucci (no un rol contratado); operar por CLI es aceptable para una sola ejecución de pre-carga.
- Postergar la web reduce el alcance del Hito 1 en al menos ~30%, dejando margen para imprevistos.

## Consecuencias

**Positivas**:

- El equipo se concentra en mobile, dominando una sola superficie de UI.
- Reduce el riesgo de no llegar al Hito 1.
- El back office web (Fase 2) se diseña con más datos reales de uso.

**Negativas**:

- El admin debe operar por CLI, lo que requiere que sea técnicamente capaz (Franco). Mitigación: script auto-documentado + CSV bien especificado.
- Si un stakeholder del Colegio quiere ver el producto desde su escritorio en la demo, no podrá. Mitigación: mirroring del celular del demo + slides; alternativa a la urgencia es mostrar el mobile en pantalla grande.
- No hay captación B2C orgánica vía SEO en el período Hito 1 → Hito 2. Mitigación: aceptado, captación es Fase 2.

**Neutras**:

- La arquitectura lógica del backend ya soporta agregar clientes web sin cambio mayor (la API es la misma).
